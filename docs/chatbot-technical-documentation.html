01

Systemüberblick

Der Chatbot ist eine clientseitige Chat-Oberfläche mit serverseitigen Next.js API-Routen. Die Antworten werden per Streaming vom LLM geliefert. Projektspezifisches Wissen wird vorab in einen lokalen JSON-Index überführt und pro Nutzerfrage semantisch abgerufen.

Client

`ChatProvider` kapselt den Chat-State, `useChat` versendet Nachrichten an `/api/chat`, und die UI rendert den AI-SDK-Stream als `UIMessage[]`.

ChatContext.tsx @ai-sdk/react UIMessage

Server API

Route Handler laden Prompts, initialisieren die Knowledge Base, erzeugen RAG-Kontext und leiten Messages an OpenRouter weiter.

/api/chat /api/suggestions /api/rag/reindex

RAG Layer

Dateien aus `src/data/knowledge-base` werden geparst, gechunkt, eingebettet und über Cosine Similarity zur Anfrage passend wieder abgerufen.

chunkTokens 512 topK 20 minSimilarity 0.3

Architektur

Die Anwendung trennt UI-State, API-Orchestrierung und RAG-Verarbeitung. Der Vector Store ist bewusst dateibasiert und benötigt keine externe Datenbank.

03

End-to-End Datenfluss

Eine Nutzerfrage durchläuft UI-State, AI-SDK-Message-Konvertierung, semantisches Retrieval, Prompt-Augmentation und schließlich den Streaming-Response-Kanal zurück zur Oberfläche.

1Eingabe`ChatInput` ruft `handleSubmit` im `ChatContext` auf.
2Message Send`useChat.sendMessage()` sendet Text und `model` an `/api/chat`.
3QueryDie letzte User-Message wird mit `extractTextFromMessage()` extrahiert.
4RetrievalRAG sucht passende Chunks zur User-Query.
5LLM Call`streamText()` ruft `openrouter.chat(selectedModel)` auf.
6Stream UI`toUIMessageStreamResponse()` aktualisiert die Chat-Messages.
const result = streamText({
  model: openrouter.chat(selectedModel),
  system: systemContext || undefined,
  messages: convertToModelMessages(messages),
});

return result.toUIMessageStreamResponse();
04

Knowledge-Verarbeitung

Die Knowledge Base wird aus lokalen Dateien aufgebaut. Der Index enthält Chunks, Embeddings und einen File-Index mit Hash und Modification Time, damit unveränderte Dateien übersprungen werden können.

Quelle und Formate

`RAG_CONFIG.knowledgeBasePath` zeigt auf `src/data/knowledge-base`. Unterstützt werden `.pdf`, `.docx`, `.txt`, `.md`, `.json` und `.csv`. Versteckte Dateien und Verzeichnisse werden ignoriert.

Change Detection

Für jede Datei wird ein SHA256-Hash berechnet und zusammen mit `mtimeMs` im `fileIndex` abgelegt. Wenn Hash und mtime unverändert sind, wird die Datei bei normalem Reindexing übersprungen.

1Scan`readdir(..., { recursive: true })` liest alle Knowledge-Einträge.
2FilterNur Dateien mit unterstützter Extension werden verarbeitet.
3Parse`parseFile()` delegiert an PDF, DOCX, Text, JSON oder CSV Parser.
4Chunk`chunkByFileType()` erzeugt tokenbasierte Chunks.
5EmbedOpenRouter Embeddings API erzeugt Vektoren.
6Persist`FileVectorStore.addChunks()` speichert den JSON-Index.
Format Parser Verhalten
.md, .txt parseText() Liest UTF-8-Text direkt aus dem Dateisystem.
.pdf parsePDF() Extrahiert Rohtext mit `pdf2json`; Dateien über 50 MB werden abgelehnt.
.docx parseDOCX() Extrahiert Rohtext mit `mammoth.extractRawText()`.
.json parseJSON() Parst JSON und serialisiert Objekte formatiert als lesbaren Text.
.csv parseCSV() Wandelt Header in `Spalten:` und Datenzeilen in nummerierte Textzeilen um. 
export const RAG_CONFIG = {
  knowledgeBasePath: join(process.cwd(), "src", "data", "knowledge-base"),
  supportedFormats: [".pdf", ".docx", ".txt", ".md", ".json", ".csv"],
  chunkTokens: 512,
  chunkOverlapTokens: 32,
  topK: 20,
  minSimilarity: 0.3,
  embeddingBatchSize: 16,
  maxFileSizeMB: 50,
};
05

Retrieval zur Laufzeit

Pro Anfrage wird die User-Query eingebettet und mit allen Chunk-Embeddings im lokalen Index verglichen. Die besten Treffer werden als Kontext in den System-Prompt eingebettet.

Similarity Search

`FileVectorStore.search()` berechnet Cosine Similarity zwischen Query-Embedding und jedem gespeicherten Chunk-Embedding. Ergebnisse unter `minSimilarity` werden verworfen und danach absteigend sortiert.

Kontextbudget

`loadRelevantContext()` begrenzt den RAG-Kontext auf ungefähr 2000 Tokens. Chunks werden in Score-Reihenfolge angehängt, bis das Budget erreicht ist. 

## Verfügbare Informationen:

### chatbot-architecture.md (Chunk 0, Relevanz: 0.82)

...relevanter Knowledge-Text...
Wenn keine Chunks gefunden werden oder der Index leer ist, liefert die RAG-Suche keinen Zusatzkontext. Bei Fehlern in `loadDocuments.ts` fällt das System auf `loadFullContext(true)` zurück, also System-Prompt plus klassische Dokumente aus `system-messages/documents`, sofern vorhanden.
06

LLM-Kommunikation

Die Anwendung nutzt OpenRouter als einheitlichen Provider für mehrere Chat-Modelle. Chat-Antworten werden gestreamt, Suggestions werden als strukturiertes Objekt erzeugt.

Chat Completion

`/api/chat` erzeugt einen OpenRouter-Client mit `OPENROUTER_API_KEY`, wählt das vom Client gesendete Modell oder `google/gemini-2.5-flash-lite` als Fallback und ruft `streamText()` auf.

Embeddings

Die RAG-Schicht ruft `https://openrouter.ai/api/v1/embeddings` direkt per `fetch` auf. Als Embedding-Modell ist `text-embedding-3-small` hinterlegt.

Message-Konvertierung

Client-Messages liegen als `UIMessage[]` vor. Vor dem Provider-Aufruf werden sie mit `convertToModelMessages(messages)` in das Modellformat des AI SDK übertragen.

Usage Logging

`onFinish` loggt `finishReason` und `usage`. Das ist aktuell Debug-/Betriebsinformation und wird nicht an den Client zurückgegeben. 

const openrouter = createOpenRouter({
  apiKey: process.env.OPENROUTER_API_KEY,
});

const selectedModel = model || "google/gemini-2.5-flash-lite";
07

API-Referenz

Die API-Routen sind Next.js Route Handler. Alle relevanten AI-Antworten laufen über den Server, damit Prompts, Knowledge und API-Key nicht im Browser liegen.

POST /api/chat

Hauptendpoint für Chat-Antworten mit RAG-Kontext und Streaming-Response.

// Request
{
  "messages": UIMessage[],
  "model": "google/gemini-2.5-flash-lite"
}

// Response
AI SDK UIMessage stream via toUIMessageStreamResponse()
POST /api/suggestions

Generiert 3 bis 5 Folgefragen auf Basis des Chatverlaufs.

// Response
{
  "suggestions": [
    "Welche Komponenten sind beteiligt?",
    "Wie wird der RAG-Index aktualisiert?"
  ]
}
GET /api/rag/reindex

Liefert den aktuellen Index-Status mit Anzahl der Chunks und Dateien.

{
  "indexed": true,
  "chunks": 42,
  "files": 4,
  "createdAt": 1710000000000,
  "updatedAt": 1710000100000
}
POST /api/rag/reindex

Startet manuelles Reindexing. Mit `{"force": true}` werden Dateien unabhängig von Hash/mtime neu verarbeitet.

curl -X POST http://localhost:3000/api/rag/reindex \
  -H "Content-Type: application/json" \
  -d '{"force": true}'
08

Konfiguration & Betrieb

Die wichtigsten Betriebsparameter liegen in Environment Variables, `constants.ts` und `RAG_CONFIG`. Der RAG-Index ist ein Cache-Artefakt und kann neu erzeugt werden.

Konfiguration Ort Bedeutung
OPENROUTER_API_KEY Environment Pflichtwert für Chat Completion und Embeddings. Fehlt der Key, antworten die API-Routen mit Fehlern.
MODELS src/lib/constants.ts Liste der im UI auswählbaren OpenRouter-Modell-IDs.
RAG_CONFIG.cachePath src/lib/rag/config.ts Lokal `.next/cache/rag-index.json`, auf Vercel `/tmp/rag-index.json`.
maxDuration API-Routen Antworten dürfen bis zu 30 Sekunden laufen.

Bekannte Einschränkungen

Der aktuelle Vector Store ist dateibasiert und linear durchsucht. Das ist einfach betreibbar, aber für sehr große Knowledge Bases nicht optimal. OCR und PII-Redaction sind in der Konfiguration als Phase-2-Optionen markiert, aber nicht aktiv.

Initialisierung

`/api/chat` startet `initializeKnowledgeBase()` beim Modul-Load nicht-blockierend. Pro Request wird maximal 5 Sekunden darauf gewartet. Falls die Initialisierung noch läuft, versucht die Route trotzdem mit vorhandenem oder fallbackfähigem Kontext fortzufahren.

09

Entwickler-Referenz

Die folgenden Dateien sind die wichtigsten Einstiegspunkte, wenn Verhalten, Knowledge, Modellwahl oder Datenverarbeitung angepasst werden sollen.

src/app/api/chat/route.ts Orchestriert Chat-Requests, wartet auf Knowledge-Initialisierung, lädt RAG-Kontext und streamt das LLM-Ergebnis.
src/lib/loadDocuments.ts Lädt System- und Suggestion-Prompts und kombiniert System-Prompt mit relevantem RAG-Kontext.
src/lib/rag/index.ts Enthält Indexierung, Embedding-Aufrufe, Suche und Kontextformatierung.
src/lib/rag/vectorStore.ts Implementiert JSON-basierten Vector Store, Persistenz und Cosine Similarity.
src/lib/rag/chunker.ts Teilt Text oder Code tokenbasiert in überlappende Chunks.
src/lib/rag/parsers.ts Extrahiert Text aus unterstützten Dateiformaten.
src/components/chatbot/ChatContext.tsx Zentraler Client-State, sendet Messages, verarbeitet Reset/Stop/Model-Wechsel und bindet Suggestions an.
src/hooks/useSuggestions.ts Triggert nach Assistant-Antworten debounced den Suggestions-Endpoint und verwaltet Abbruchzustände.

Typen

Zentrale RAG-Typen sind `Chunk`, `ChunkMetadata`, `SearchResult`, `RelevantContext`, `VectorStoreIndex` und `FileIndexEntry`. Sie definieren die Struktur des gespeicherten Indexes und der Suchergebnisse.

Erweiterungspunkte

Neue Dateiformate werden über `supportedFormats` und `parseFile()` ergänzt. Neue Vector Stores können hinter dem `IVectorStore`-Interface umgesetzt werden. Neue Chat-Modelle werden in `MODELS` ergänzt.
